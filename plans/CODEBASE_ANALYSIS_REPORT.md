# Codebase Analysis Report

## Project Summary
- Project type: multi-client EV charging platform monorepo (Spring Boot backend + React admin/owner web + Android app).
- Primary operational model: centralized backend API with JWT auth, PostgreSQL persistence, and WebSocket telemetry for charging simulation/live updates.
- File inventory analyzed: 207 project files (`backend`: 75, `web`: 31, `android`: 87, docs/log/meta: 14).
- Raw directory also contains generated/vendor trees (`.git`, `web/node_modules`, `backend/target`); this report focuses on project-owned source/config/docs/log files represented in `file_summary.csv`.

## Tech Stack
- Backend: Java 21, Spring Boot 3.3.5, Spring Web/Data JPA/Security/WebSocket, PostgreSQL driver, jjwt 0.12.5, Stripe Java SDK 25.2.0, Maven wrapper.
- Web: React 19 + Vite 6, React Router 7, Redux Toolkit, Axios, Tailwind CSS 4, MUI, Recharts, Leaflet/React-Leaflet, Framer Motion.
- Android: Kotlin + Jetpack Compose, Navigation Compose, Retrofit + Gson + OkHttp logging, DataStore, Coroutines, Google Maps Compose + Play Services, custom STOMP client.
- Infra/external services: PostgreSQL (Supabase-compatible config), Stripe payments, WebSocket/SockJS, Nominatim/OpenStreetMap geocoding/tiles, Google Maps API.

## Architecture Overview (with diagram)
Overall pattern is layered monolith backend with two frontend clients. Backend modules are organized by classic controller -> service -> repository -> entity layers with cross-cutting auth/security and websocket telemetry.

```text
             +-----------------------------+        +-----------------------------+
             | React Web (admin/owner)    |        | Android App (end users)     |
             | Browser + Redux + Axios    |        | Compose + ViewModels        |
             +-------------+---------------+        +--------------+--------------+
                           | REST (JWT)                          | REST (JWT)
                           +--------------------+----------------+
                                                |
                                   +------------v-------------+
                                   | Spring Boot Backend       |
                                   | Controllers               |
                                   | Services (business rules) |
                                   | Repositories (JPA)        |
                                   +------+-----------+--------+
                                          |           |
                              +-----------v--+     +--v------------------+
                              | PostgreSQL   |     | WebSocket/STOMP     |
                              | Stations,    |     | Telemetry topics    |
                              | bookings,... |     | (simulation/live)   |
                              +--------------+     +---------------------+
                                          |
                         +----------------+-------------------+
                         | External Integrations              |
                         | Stripe API, Nominatim, Google Maps|
                         +------------------------------------+
```

## Module / Component Breakdown
- Backend entry point: `backend/src/main/java/com/ganesh/EV_Project/EvProjectApplication.java`.
- Backend controllers: auth, stations, dispensaries, charger slots, bookings, charging sessions, IoT, payments, recommendations, websocket demo.
- Backend services: booking lifecycle/expiry, charging simulator scheduler, station recommendation scoring, payment/Stripe orchestration, OTP/user/station/dispensary/slot services.
- Backend persistence: Spring Data repositories over JPA entities (`User`, `Station`, `Dispensary`, `ChargerSlot`, `Booking`, `ChargingSession`, `Payment`, `Otp`, `IoTSensorData`).
- Web entry point: `web/src/main.jsx`; route composition in `web/src/App.jsx` with role-based guards (`PrivateRoute`, `RoleRoute`) and dashboard layout shell.
- Android entry point: `android/app/src/main/java/com/ganesh/ev/MainActivity.kt`; Compose NavHost routes login, map discovery, booking, charging, history, profile.
- External dependencies in code/config: PostgreSQL/Supabase URL patterns, Stripe keys/webhook secret, Google Maps API key, OpenStreetMap/Nominatim calls, WebSocket SockJS endpoint `/ws`.

## Data Flow
### 1) Authentication / session bootstrap
1. Client sends OTP/login/register request to `/api/auth/*`.
2. `AuthController` validates input and delegates to OTP/user services/repositories.
3. `JwtUtil` mints token; response returns user + token.
4. Web stores token in Redux + localStorage; Android stores session in DataStore and sets Retrofit auth header.

### 2) Station discovery and booking lifecycle
1. Client requests stations/viewport-nearby endpoints (`StationController` + `StationRecommendationController`).
2. Recommendation service filters by viewport, ranks by distance/availability/price factors, and returns nearby detail + marker pins.
3. User creates booking via `/api/bookings`; `BookingService` validates user/slot/station state, persists `Booking`, and updates slot status.
4. Background scheduler in `BookingService` expires stale unstarted bookings and reopens slots.

### 3) Charging + telemetry + payment
1. Client starts charging via `/api/charging/start`; backend creates session and starts simulation tracking.
2. `ChargingSimulatorService` scheduled tick updates session telemetry and pushes updates on websocket topics.
3. Client listens through STOMP client (`/topic/...`) and/or polls live power endpoint.
4. On stop, backend finalizes session and exposes payment intent creation (`/api/payments/create-intent/{bookingId}`) through Stripe.
5. Payment webhook/status updates persist payment outcomes and booking payment status.

## Key Patterns & Design Decisions
- Layered architecture in backend with clear controller/service/repository boundaries.
- DTO-driven API boundaries (`APIResponse`, request DTOs, station recommendation DTOs).
- JWT stateless auth with custom request filter in Spring Security chain.
- Scheduled jobs for lifecycle management (`expireUnstartedBookings`) and telemetry simulation ticks.
- Hybrid real-time model: WebSocket push + REST polling fallback in Android viewmodels.
- Compose + ViewModel + `StateFlow` pattern on Android; React router + guard components on web.
- Config-driven runtime values (`BuildConfig.BASE_URL`, Spring properties, env placeholders) with current tendency toward dev defaults in committed files.

## Things to Watch Out For
- Security exposure: test password bypass in `AuthController` should be removed outside local dev.
- Secret hygiene: committed credentials/API keys in `application-dev.properties` and `android/local.properties` must be rotated and externalized.
- Access control posture: permissive CORS and broad `permitAll` URL patterns weaken backend boundary protections.
- Role-check consistency: enum/string comparisons in booking authorization logic are fragile and should be normalized.
- State management drift: web `stationSlice` exists but is not registered in store (possible dead code or incomplete migration).
- API response inconsistency: some endpoints return wrapped `APIResponse`, others raw entities, increasing client parsing complexity.
- Large files (`ManageStationPage.jsx`, `HomeScreen.kt`, `StationDetailScreen.kt`, `BookingDetailScreen.kt`) have high cognitive load and should be incrementally split.
- Historical logs show schema-evolution and compile regressions; migration discipline and CI checks should be tightened.

## Recommended Starting Points for a New Developer
1. Read `README.md` sections for architecture + setup, then verify against current code state.
2. Backend startup path: `EvProjectApplication` -> `SecurityConfig` -> `AuthController` -> `BookingService` -> repositories/entities.
3. Web shell/navigation path: `web/src/main.jsx` -> `web/src/App.jsx` -> `DashboardLayout` + route guards.
4. Android flow path: `MainActivity.kt` -> key viewmodels (`AuthViewModel`, `StationViewModel`, `BookingViewModel`, `ChargingViewModel`) -> `ApiService.kt`.
5. Cross-cutting integrations to understand early: JWT claims/token usage, station/booking status transitions, websocket topic payloads, and Stripe payment state transitions.
6. Before feature work, clean local secret handling and remove test-security shortcuts to reduce accidental production risk.
